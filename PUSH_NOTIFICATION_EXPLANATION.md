# Push Notifications - Complete Technical Explanation

## 🔴 IMPORTANT: VAPID Keys vs Expo Push Tokens

**These are TWO completely different systems:**

### ❌ VAPID Keys (NOT for your mobile app)
**What they are**: Keys for **Web Push Protocol** - sending notifications to web browsers
**Where they work**: 
- ✅ Safari, Chrome, Firefox (browsers running in web browsers)
- ❌ NOT for React Native apps
- ❌ NOT for Expo mobile apps

**What VAPID does**:
- Used to authenticate with browser push services (Firefox's server, Google's FCM for browsers, etc.)
- Allows you to send notifications to browsers via JavaScript ServiceWorkers
- Protocol: Browser → Your server → Browser's push service → Browser

### ✅ Expo Push Tokens (THIS IS WHAT YOU USE)
**What they are**: Tokens generated by **Expo Push Notification Service** specifically for mobile apps
**Where they work**:
- ✅ React Native apps (using expo-notifications)
- ✅ Android (via Firebase Cloud Messaging)
- ✅ iOS (via Apple Push Notification service)
- ✅ Since you're using Expo, this is YOUR system

**What Expo tokens do**:
- Device registers with Expo service to get token: `ExponentPushToken[abc123...]`
- Token identifies that specific device to receive notifications
- Server sends notifications to Expo service WITH that token
- Expo service delivers to actual device via Firebase/APNS

## 📱 EXPO PUSH NOTIFICATION FLOW (What You're Actually Using)

### Step-by-Step Process

```
┌──────────────────────────────────────────────────────────────────┐
│ STEP 1: DEVICE REGISTERS FOR NOTIFICATIONS (On App Launch)      │
└──────────────────────────────────────────────────────────────────┘

[Mobile App]
    │
    ├─→ import { getExpoPushTokenAsync } from 'expo-notifications'
    │
    ├─→ Call: const token = await getExpoPushTokenAsync()
    │   Result: "ExponentPushToken[WK47jNQM9WshB4YS1RnXg]"
    │
    ├─→ Store locally: AsyncStorage.setItem('expo_push_token', token)
    │
    ├─→ Send to server: POST /api/notifications/subscribe
    │   Body: { push_token: "ExponentPushToken[...]", platform: "android" }
    │
∨∨∨ SERVER RECEIVES TOKEN ∨∨∨
    │
    ├─→ Server saves to database: push_subscriptions table
    │   Row: { user_id: '123', push_token: 'ExponentPushToken[...]', is_active: true }
    │
    └─→ ✅ NOW device is subscribed!


┌──────────────────────────────────────────────────────────────────┐
│ STEP 2: SERVER SENDS NOTIFICATION (When ride assigned, etc)     │
└──────────────────────────────────────────────────────────────────┘

[Backend Server]
    │
    ├─→ Event happens: Driver gets new ride
    │
    ├─→ Query database: 
    │   SELECT push_token FROM push_subscriptions 
    │   WHERE user_id = '123' AND is_active = true
    │   Result: ['ExponentPushToken[WK47jNQM9WshB4YS1RnXg]', 'ExponentPushToken[XYZ789...]']
    │
    ├─→ For each token, prepare payload:
    │   {
    │     to: 'ExponentPushToken[WK47jNQM9WshB4YS1RnXg]',
    │     sound: 'default',
    │     title: 'New Ride Request',
    │     body: 'Pickup at Main Street',
    │     data: { rideId: '999', action: 'ride_request' }
    │   }
    │
    ├─→ Send to Expo service:
    │   POST https://exp.host/--/api/v2/push/send
    │
    └─→ ✅ Expo service receives notification


┌──────────────────────────────────────────────────────────────────┐
│ STEP 3: EXPO SERVICE DELIVERS TO DEVICE                        │
└──────────────────────────────────────────────────────────────────┘

[Expo Service]
    │
    ├─→ It knows what "ExponentPushToken[WK47jNQM9WshB4YS1RnXg]" means
    │   (it issued this token when device first registered)
    │
    ├─→ Routes notification to appropriate service:
    │   - Android: Firebase Cloud Messaging (FCM)
    │   - iOS: Apple Push Notification (APN)
    │
    └─→ ✅ Service delivers to actual device


┌──────────────────────────────────────────────────────────────────┐
│ STEP 4: DEVICE RECEIVES & SHOWS NOTIFICATION                   │
└──────────────────────────────────────────────────────────────────┘

[Mobile Device]
    │
    ├─→ Notification received by OS (Android/iOS)
    │
    ├─→ Three possible states:
    │
    │   📱 APP IN FOREGROUND:
    │   ├─→ Your expo-notifications listener fires
    │   └─→ You decide: show banner, play sound, increment badge
    │
    │   📲 APP IN BACKGROUND:
    │   ├─→ OS shows notification in notification center
    │   └─→ User can tap to open app
    │
    │   ☠️ APP CLOSED/TERMINATED:
    │   ├─→ OS shows notification on lock screen
    │   └─→ User can tap to open app
    │
    └─→ ✅ User sees notification!


┌──────────────────────────────────────────────────────────────────┐
│ STEP 5: USER TAPS NOTIFICATION (Optional)                       │
└──────────────────────────────────────────────────────────────────┘

[Mobile App - Responds to tap]
    │
    ├─→ Notification response listener fires
    │
    ├─→ Extract data: rideId, action, etc.
    │
    ├─→ Handle action: 
    │   "Accept ride" → Navigate to acceptance screen
    │   "View message" → Jump to chat
    │
    └─→ ✅ User interaction completed!
```

## 🗄️ DATABASE STRUCTURE (No RLS needed - Custom Auth)

```sql
-- Simple, no RLS required
CREATE TABLE push_subscriptions (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,              -- Your user ID (from custom auth)
  push_token TEXT NOT NULL,           -- "ExponentPushToken[...]"
  platform TEXT NOT NULL,             -- 'ios' or 'android'
  subscribed_at TIMESTAMP,
  is_active BOOLEAN DEFAULT true,
  
  UNIQUE(user_id, push_token)         -- One subscription per device
);
```

**What each column means:**
- `user_id`: The user who owns this notification subscription
- `push_token`: The ACTUAL subscription identifier (issued by Expo)
- `platform`: Which OS this device runs
- `is_active`: Whether user enabled notifications (toggle switch)

## 🔑 YOUR VAPID KEYS (Don't Panic!)

You have VAPID keys in env files because:
1. **Someone** configured them for web push (browsers)
2. But you're NOT using them for mobile notifications
3. You CAN safely ignore them for the mobile app

If in future you want to send notifications to **web users**, THEN you'd use VAPID keys.

## 📋 WHAT HAPPENS AT EACH STEP

### When user opens app (Authentication):
```
1. User logs in
2. Auth context calls: subscribeToPushNotifications()
3. Mobile requests notification permission: "Allow notifications?"
4. User grants permission
5. Gets Expo token: await getExpoPushTokenAsync()
6. Sends to server: POST /api/notifications/subscribe
7. Server saves to database
8. ✅ User now subscribed
```

### When user toggles OFF in settings:
```
1. User taps toggle switch
2. Hook calls: DELETE /api/notifications/subscribe
3. Server marks: is_active = false
4. ✅ User no longer gets notifications (this device)
```

### When user toggles OFF then ON again:
```
1. User taps toggle switch again
2. Hook calls: POST /api/notifications/subscribe
3. Server marks: is_active = true
4. ✅ User subscribed again
```

### When admin sends ride notification:
```
1. Admin creates/assigns ride
2. Backend queries: SELECT push_token FROM push_subscriptions 
   WHERE user_id = '123' AND is_active = true
3. For each token found, calls Expo API
4. Expo routes to Firebase/APNS
5. Device gets notification ✅
```

## 🎯 WHAT IS A PUSH TOKEN?

**It's like a mailing address for your phone.**

Think of it this way:
- Your phone = House
- Push token = Your home address
- Expo service = Post office
- Server = Someone sending you a letter

```
Server says: "Send this notification to ExponentPushToken[WK47...]"
     ↓
Expo service thinks: "Ah, that token = Device XYZ123 running on Android"
     ↓
Sends to Firebase/APNS
     ↓
Firebase/APNS routes to actual device
     ↓
Device receives: "You have a new ride!"
```

**That token is unique per device.** Same user on two devices = two different tokens.

## 🛠️ HOW IT WORKS IN CODE

### Step 1: Get Token (Expo generates it)
```tsx
// This is automatic, but shown for clarity
import * as Notifications from 'expo-notifications';

const token = await Notifications.getExpoPushTokenAsync();
console.log(token.data); // "ExponentPushToken[abc123...]"
```

### Step 2: Send Token to Server
```tsx
// Mobile app sends its token to your backend
const response = await fetch('/api/notifications/subscribe', {
  method: 'POST',
  body: JSON.stringify({
    push_token: token.data,  // "ExponentPushToken[...]"
    platform: Platform.OS    // 'android' or 'ios'
  })
});
```

### Step 3: Server Stores Token
```tsx
// Backend saves to Supabase
await supabase.from('push_subscriptions').insert({
  user_id: userId,
  push_token: 'ExponentPushToken[abc123...]',  // ← This is the subscription ID
  platform: 'android',
  is_active: true
});
```

### Step 4: Server Sends Notification
```tsx
// When something happens (ride assigned)
const { data: subscriptions } = await supabase
  .from('push_subscriptions')
  .select('push_token')
  .eq('user_id', driverId)
  .eq('is_active', true);

// Send to each token
for (const sub of subscriptions) {
  await fetch('https://exp.host/--/api/v2/push/send', {
    method: 'POST',
    body: JSON.stringify({
      to: sub.push_token,  // "ExponentPushToken[...]"
      title: 'New Ride',
      body: 'Pickup at Main St',
      sound: 'default'
    })
  });
}
```

### Step 5: Device Receives
```tsx
// Mobile app has listeners configured
Notifications.addNotificationReceivedListener(notification => {
  // Handle foreground notification
  console.log('Notification received:', notification.request.content.title);
});

Notifications.addNotificationResponseReceivedListener(response => {
  // Handle user tapping notification
  console.log('User tapped:', response.notification.request.content.title);
});
```

## 📌 SUMMARY

1. **VAPID Keys** = For web browsers (not you)
2. **Expo Push Tokens** = For your mobile app (THIS IS YOU) ✅
3. **Phone registers** = Gets token from Expo
4. **Token saved** = In database for future use
5. **Send notification** = Server queries tokens, sends to Expo
6. **Expo delivers** = Routes to Firebase/APNS → Device
7. **Device shows** = User sees notification ✅

Your system: **Mobile → Expo Service → Firebase/APNS → Device**

(VAPID would be for: **Mobile Web → Your Server → Browser Push Service → Browser**, but you're not doing this)
